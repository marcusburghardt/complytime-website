# Feature Specification: Skip Lockfile-Only Automated PRs

**Feature ID**: `014-skip-lockfile-only-pr`
**Phase**: 3 (CI/CD Improvements)
**Created**: 2026-04-09
**Status**: Done (retroactive spec)
**Input**: Review comment by @marcusburghardt on complytime/website PR #8

## Overview

The weekly `sync-content-check.yml` workflow scanned upstream repositories for new commits, updated `.content-lock.json` with the latest branch SHAs, and unconditionally opened an automated pull request — even when every upstream doc file was byte-for-byte identical to what the site would already render at the new SHA. PR #8 demonstrated this: 5 repos newly added to the lockfile, 32 files unchanged, PR opened.

This feature gates automated PR creation on whether at least one doc file actually changes, and fixes `syncResult.hasChanges()` to include file-level changes in addition to repo-level project-page changes.

**Dependencies**: Go 1.25+ (sync tool), GitHub Actions (`sync-content-check.yml`); no new dependencies added.

## Scope

### In Scope

| ID | Capability |
|----|-----------|
| IS-001 | Two-step workflow: `--update-lock` step writes new lock SHAs; `--write` step syncs doc content with the updated lock and captures `has_changes` output |
| IS-002 | PR creation guarded by `if: steps.check.outputs.has_changes == 'true'` — skipped when all docs are unchanged |
| IS-003 | `syncResult.hasChanges()` returns `true` when `changedRepoFiles` contains at least one written file, in addition to the existing `added`/`updated`/`removed` checks |
| IS-004 | `TestHasChanges` table-driven test (7 subtests) covering the new `changedRepoFiles` code path |

### Out of Scope

- Committing content files into the PR branch (content is gitignored and synced at deploy time)
- File-level comparison without fetching content (would require a new `--check-files` flag)
- Changes to the deploy workflow or lockfile approval semantics

### Edge Cases

| Case | Expected Behavior |
|------|-------------------|
| Lock SHA unchanged for all repos | `writeFileSafe` writes nothing; `has_changes=false`; no PR |
| New repo added to lock, docs already in sync | All writes skipped; `has_changes=false`; no PR |
| README unchanged but doc sub-pages changed | `changedRepoFiles` populated; `has_changes=true`; PR opened |
| `GITHUB_OUTPUT` not set (local run) | `writeGitHubOutputs` is a no-op; behaviour unchanged |

## Requirements

### Functional Requirements

- **FR-001**: The workflow MUST update `.content-lock.json` in a dedicated `--update-lock` step before running the content check.
- **FR-002**: The workflow MUST run a `--write` content sync with the updated lockfile and capture `has_changes` from its GitHub Actions output.
- **FR-003**: The "Create or update PR" step MUST be guarded by `if: steps.check.outputs.has_changes == 'true'`.
- **FR-004**: `syncResult.hasChanges()` MUST return `true` when `changedRepoFiles` contains at least one non-empty file slice, regardless of whether `added`/`updated`/`removed` are empty.
- **FR-005**: The PR body summary MUST be generated by the `--write` step so it reflects actual file changes.

### Non-Functional Requirements

- **NFR-001**: Two sync-tool invocations per weekly run is acceptable given the cron cadence and 3-minute timeout.
- **NFR-002**: Written `content/docs/projects/*/` files must not be staged into the PR branch (they are gitignored).

## Success Criteria

| ID | Criterion |
|----|-----------|
| SC-001 | PR creation is skipped when all docs are unchanged (PR #8 scenario reproduced and fixed) |
| SC-002 | `go test ./cmd/sync-content/... -run TestHasChanges` passes all 7 subtests |
| SC-003 | `go test -race ./cmd/sync-content/...` passes with zero data race warnings |
| SC-004 | PR body accurately describes which files changed, generated from the `--write` step |

## Implementation Notes

The `content/docs/projects/*/` path is gitignored — content written by `--write` lives only on the runner's ephemeral disk and is never staged. The `add-paths: .content-lock.json` in the PR step ensures only the lockfile reaches the PR branch. Running the tool twice per cron execution (once for lock update, once for content check) doubles the API calls but is within the tool's authenticated rate-limit budget.
